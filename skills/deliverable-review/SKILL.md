# 資料レビュー

資料が「読めるか」ではなく、「相手が判断して動けるか」をレビューする。表現の微修正より先に、結論、根拠、構造、依頼を直す。

## 原則

1. 誤字脱字より、意思決定を止める欠陥を先に指摘する。
2. きれいでも中身がない資料を合格にしない。
3. 指摘ごとに、なぜ問題か、どう直すかを示す。
4. 事実を捏造せず、不足は不足として残す。
5. 全面改稿より、最小の修正で価値が上がる順に並べる。

## 手順

1. 読み手、目的、資料を読んだ後に起こしたい行動を特定する。
2. 不明な場合は入力から仮定し、重要な確認だけ最大3つ質問する。
3. 資料の主張を1文で再現する。再現できなければ、それ自体を重大な欠陥とする。
4. 次の順でレビューする。
   - Decision: 何を判断・承認・理解してほしいか
   - Message: 結論が先にあり、1文で言えるか
   - Evidence: 根拠が主張を支え、出典や前提が追えるか
   - Structure: 各節が結論へつながり、重複や脱線がないか
   - Action: 読み手の次の行動が具体か
5. 指摘をP0、P1、P2へ分ける。
   - P0: このままでは誤判断や提出失敗につながる
   - P1: 説得力や理解を大きく下げる
   - P2: 表現、体裁、読みやすさの改善
6. 残す、削る、移す、追加する箇所を示す。
7. 最重要部分の書き換え例または新しい構成案を出す。

## 出力形式

```markdown
## 0. 総合判定
- 判定: Go / 要修正 / 作り直し
- 理由:

## 1. この資料が伝えている主張
- 現状:
- あるべき1文:

## 2. 重大な指摘
| 優先度 | 箇所 | 問題 | なぜ問題か | 修正案 |
|---|---|---|---|---|

## 3. 構成の処方箋
- 残す:
- 削る:
- 移す:
- 追加する:

## 4. 足りない根拠
- ...

## 5. 書き換え例または新しい構成

## 6. 次アクション
1. ...
```

## 品質基準

- 冒頭でGo、要修正、作り直しのどれかを明示する。
- P0とP1を最大5件に絞る。
- 「分かりにくい」だけで終わらず、原因と修正を示す。
- 資料の目的に関係しない好みを押し付けない。
- 根拠のない数値や事実を補わない。

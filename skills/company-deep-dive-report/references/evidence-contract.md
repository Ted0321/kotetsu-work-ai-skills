# 証拠契約

## 1. 証拠の優先順位

| Tier | 例 | 主な用途 |
|---|---|---|
| A | 法定開示、決算資料、規制当局、公式統計、契約・特許・裁判資料 | 数値、戦略、権利、規制の確定 |
| B | 会社公式発表、製品仕様、経営者発言、取引先の一次発表 | 施策、製品、関係性の確認 |
| C | 信頼できる報道、専門調査、学術資料、業界団体 | 文脈、比較、外部評価 |
| D | レビュー、SNS、求人、検索トレンド、コミュニティ | 需要や組織の補助信号 |

Tier Dを単独で市場規模、業績、製品性能の証明に使わない。

## 2. 証拠台帳

`evidence_ledger.csv` は次の列を維持する。

| 列 | 内容 |
|---|---|
| evidence_id | `E001`形式の一意ID |
| collected_at | 取得日時とタイムゾーン |
| published_at | 公開日。不明なら空欄 |
| source_tier | A / B / C / D |
| source_type | filing、earnings、news、review等 |
| publisher | 発行主体 |
| title | 文書・ページ名 |
| url_or_path | URLまたはローカルパス |
| geography | 対象地域 |
| period | 対象期間 |
| evidence | 引用しすぎず、検証可能な要点または数値 |
| supports | 支持する主張ID。未接続なら空欄 |
| contradicts | 反証する主張ID。なければ空欄 |
| limitations | 定義差、偏り、欠測など |

転載記事、同一プレスリリース由来の記事、同一データセットの再掲には同じ `source_family` と見なせる記述を `limitations` に入れる。

## 3. 主張グラフ

各主張には次を持たせる。

```json
{
  "claim_id": "C001",
  "tier": 1,
  "classification": "INFERENCE",
  "statement": "対象企業固有の検証可能な主張",
  "because": "証拠から主張に至る因果",
  "implication": "意思決定への意味",
  "confidence": "MEDIUM",
  "evidence_ids": ["E001", "E002"],
  "counterevidence_ids": ["E003"],
  "falsifier": "何が観測されたら主張を棄却するか",
  "open_questions": []
}
```

### 分類

- `FACT`: ソースが直接述べる確認可能な事実。
- `INFERENCE`: 複数の事実を因果的に解釈した主張。
- `HYPOTHESIS`: 有力だが追加検証が必要な説明。
- `UNKNOWN`: 意思決定上重要だが確認できない項目。

## 4. 確度判定

### HIGH

- 独立した複数ソースが整合する。
- 一次情報があり、定義と期間が明確である。
- 強い反証がなく、因果の飛躍が小さい。

### MEDIUM

- 根拠はあるが、一部が二次情報、推定、古い情報である。
- 代替説明が残る、または地域・期間の外挿がある。

### LOW

- 単一ソース、間接信号、未検証の推定に依存する。
- 重要な反証または定義不明が残る。

確度は主張の重要性ではなく、証拠の質と推論距離で決める。

## 5. 矛盾処理

1. 数値定義、対象範囲、期間、通貨、調整前後を確認する。
2. 同じ定義なら、公開日、一次性、収集方法を比較する。
3. 解消できなければ両方を残し、どの判断がどう変わるか示す。
4. 矛盾を平均値や曖昧な文章で消さない。

## 6. 引用とトレーサビリティ

- レポート本文の重要主張末尾に `[C001 | E001,E002]` の形でIDを置く。
- Web出典は読者が開けるリンクを併記する。
- PDFは可能ならページ番号を記録する。
- 取得時点で変わり得る情報には取得日時を付ける。
- ソースにない数字、引用、因果を作らない。
